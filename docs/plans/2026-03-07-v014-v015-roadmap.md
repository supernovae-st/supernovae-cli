## Part 1: Socratic Review — Questioning Our Assumptions

Before planning features, let's step back and question everything.

### The Fundamental Questions

```
╭─────────────────────────────────────────────────────────────────────────────╮
│                                                                             │
│  Q1: Who is our user?                                                       │
│                                                                             │
│      A developer building AI-powered applications.                          │
│      They use Claude Code, VS Code, or Cursor.                              │
│      They want AI workflows without complexity.                             │
│                                                                             │
│  Q2: What job are they hiring spn to do?                                    │
│                                                                             │
│      "Help me set up and manage my AI development environment              │
│       so I can focus on building, not configuring."                        │
│                                                                             │
│  Q3: What's the emotional journey we want?                                  │
│                                                                             │
│      😕 Confused    →  "What tools do I need?"                              │
│      🤔 Curious     →  "This looks organized..."                            │
│      😊 Confident   →  "I know exactly what to do"                          │
│      🚀 Empowered   →  "I can build anything!"                              │
│                                                                             │
╰─────────────────────────────────────────────────────────────────────────────╯
```

## Part 2: Design Principles

Based on our Socratic review, here are the principles guiding v0.14 and v0.15:

```
╔═════════════════════════════════════════════════════════════════════════════╗
║                         DESIGN PRINCIPLES                                   ║
╠═════════════════════════════════════════════════════════════════════════════╣
║                                                                             ║
║  1. GUIDE, DON'T GATEKEEP                                                   ║
║     ─────────────────────                                                   ║
║     When user forgets an argument, help them choose.                        ║
║     Don't show error, show options.                                         ║
║                                                                             ║
║     ❌ error: required argument missing                                     ║
║     ✅ ? Which provider would you like to configure?                        ║
║                                                                             ║
║  2. SHOW PROGRESS, REDUCE ANXIETY                                           ║
║     ─────────────────────────────                                           ║
║     Long operations should show what's happening.                           ║
║     Users shouldn't wonder "is it stuck?"                                   ║
║                                                                             ║
║     ❌ [cursor blinking for 30 seconds]                                     ║
║     ✅ Downloading llama3.2 [████████░░] 45% ETA: 2m                       ║
║                                                                             ║
║  3. ONE PATTERN TO RULE THEM ALL                                            ║
║     ─────────────────────────────                                           ║
║     Reduce cognitive load with consistent syntax.                           ║
║     Users learn once, apply everywhere.                                     ║
║                                                                             ║
║     ❌ spn mcp add X / spn skill add Y / spn model pull Z                  ║
║     ✅ spn add @mcp/X / spn add @skills/Y / spn add @models/Z              ║
║                                                                             ║
║  4. CELEBRATE SUCCESS, EXPLAIN FAILURE                                      ║
║     ────────────────────────────────                                        ║
║     Make wins feel good. Make errors actionable.                            ║
║                                                                             ║
║     ❌ Done.                                                                ║
║     ✅ ✓ Added @mcp/neo4j — run `spn sync` to update your editor           ║
║                                                                             ║
║  5. PROGRESSIVE DISCLOSURE                                                  ║
║     ──────────────────────                                                  ║
║     Simple by default, powerful when needed.                                ║
║     Don't overwhelm beginners, don't limit experts.                         ║
║                                                                             ║
║     Level 1: spn add @mcp/neo4j                                            ║
║     Level 2: spn add @mcp/neo4j --scope=global --config=./neo4j.yaml       ║
║     Level 3: spn add @mcp/neo4j@2.1.0 --scope=global --dry-run             ║
║                                                                             ║
╚═════════════════════════════════════════════════════════════════════════════╝
```

### 4.4 Feature: Smart Error Messages

**The Problem:**
```
$ spn provider get anthropic
Error: Key not found in keychain
```

**The Solution:**
```
$ spn provider get anthropic

  ⚠️  No API key found for anthropic

  The key might be:
  • Not set yet
  • Stored in environment variable ANTHROPIC_API_KEY
  • In a .env file (less secure)

  To fix this:
    spn provider set anthropic    Store key in secure keychain
    spn provider migrate          Move env vars to keychain

  Need help? Run: spn topic providers
```

### 4.5 Feature: Command Aliases

**The Problem:**
```
User must type full command names every time.
spn provider list → 14 characters
```

**The Solution:**
```
$ spn p l          # Same as: spn provider list
$ spn m p llama    # Same as: spn model pull llama
$ spn d            # Same as: spn doctor

Aliases are shown in help:
  provider, p    Manage API keys
  model, m       Manage local LLM models
  doctor, d      System diagnostic
```

**Common Aliases:**

## Part 5: v0.15.0 — "The Unification Release"

**Theme:** One mental model, infinite possibilities.

### 5.1 The Unified @ Syntax

**Current State (Cognitive Overload):**
```
# User must remember 4 different patterns:
spn mcp add neo4j              # MCP servers
spn skill add brainstorming    # Skills
spn model pull llama3.2        # Models
spn add @nika/generate-page    # Packages

# Different verbs: add, pull
# Different structures: namespace first vs @ prefix
```

**Unified State (One Pattern):**
```
# One pattern for everything:
spn add @mcp/neo4j
spn add @skills/brainstorming
spn add @models/llama3.2
spn add @nika/generate-page

# Always: spn add @namespace/name
# Optional version: spn add @mcp/neo4j@1.2.0
```

### A.1 ComponentRef Parser

```rust
// crates/spn/src/component.rs

use std::str::FromStr;

#[derive(Debug, Clone, PartialEq)]
pub enum Namespace {
    Mcp,
    Skills,
    Models,
    Nika,
    Novanet,
    Workflows,
    Jobs,
}

#[derive(Debug, Clone)]
pub struct ComponentRef {
    pub namespace: Namespace,
    pub name: String,
    pub version: Option<String>,
}

impl FromStr for ComponentRef {
    type Err = ComponentParseError;

    fn from_str(s: &str) -> Result<Self, Self::Err> {
        // Must start with @
        let s = s.strip_prefix('@')
            .ok_or(ComponentParseError::MissingPrefix)?;

        // Split namespace/name@version
        let (namespace_name, version) = match s.split_once('@') {
            Some((ns_name, ver)) => (ns_name, Some(ver.to_string())),
            None => (s, None),
        };

        // Split namespace/name
        let (namespace_str, name) = namespace_name.split_once('/')
            .ok_or(ComponentParseError::MissingSlash)?;

        let namespace = match namespace_str {
            "mcp" => Namespace::Mcp,
            "skills" => Namespace::Skills,
            "models" => Namespace::Models,
            "nika" => Namespace::Nika,
            "novanet" => Namespace::Novanet,
            "workflows" => Namespace::Workflows,
            "jobs" => Namespace::Jobs,
            _ => return Err(ComponentParseError::UnknownNamespace(namespace_str.to_string())),
        };

        Ok(ComponentRef {
            namespace,
            name: name.to_string(),
            version,
        })
    }
}

#[derive(Debug, thiserror::Error)]
pub enum ComponentParseError {
    #[error("Component must start with '@'")]
    MissingPrefix,
    #[error("Component must have format '@namespace/name'")]
    MissingSlash,
    #[error("Unknown namespace: {0}. Valid: @mcp/, @skills/, @models/, @nika/")]
    UnknownNamespace(String),
}
```

```rust
use indicatif::{ProgressBar, ProgressStyle, MultiProgress};
use std::time::Duration;

pub struct DownloadProgress {
    bar: ProgressBar,
}

impl DownloadProgress {
    pub fn new(total_bytes: u64, name: &str) -> Self {
        let bar = ProgressBar::new(total_bytes);
        bar.set_style(
            ProgressStyle::default_bar()
                .template("  {msg}\n  {bar:40.cyan/dim} {percent:>3}% {bytes}/{total_bytes} {eta}")
                .unwrap()
                .progress_chars("━━░")
        );
        bar.set_message(format!("Downloading {}", name));
        Self { bar }
    }

    pub fn set_position(&self, bytes: u64) {
        self.bar.set_position(bytes);
    }

    pub fn finish_success(&self, message: &str) {
        self.bar.finish_with_message(format!("✓ {}", message));
    }

    pub fn finish_error(&self, message: &str) {
        self.bar.abandon_with_message(format!("✗ {}", message));
    }
}

pub struct Spinner {
    bar: ProgressBar,
}

impl Spinner {
    pub fn new(message: &str) -> Self {
        let bar = ProgressBar::new_spinner();
        bar.set_style(
            ProgressStyle::default_spinner()
                .template("  {spinner:.cyan} {msg}")
                .unwrap()
                .tick_strings(&["⠋", "⠙", "⠹", "⠸", "⠼", "⠴", "⠦", "⠧", "⠇", "⠏"])
        );
        bar.set_message(message.to_string());
        bar.enable_steady_tick(Duration::from_millis(80));
        Self { bar }
    }

    pub fn finish_success(&self, message: &str) {
        self.bar.finish_with_message(format!("✓ {}", message));
```

```rust
// crates/spn/src/prompts.rs

use dialoguer::{theme::ColorfulTheme, FuzzySelect, Password, Confirm};
use crate::error::Result;

pub fn select_provider() -> Result<String> {
    let providers = [
        ("anthropic", "Claude API (sk-ant-...)"),
        ("openai", "GPT-4, etc. (sk-proj-...)"),
        ("mistral", "Mistral AI"),
        ("groq", "Groq (fast inference)"),
        ("deepseek", "DeepSeek"),
        ("gemini", "Google Gemini"),
    ];

    let items: Vec<String> = providers
        .iter()
        .map(|(name, desc)| format!("{:<12} {}", name, desc))
        .collect();

    let selection = FuzzySelect::with_theme(&ColorfulTheme::default())
        .with_prompt("Which provider would you like to configure?")
        .items(&items)
        .default(0)
        .interact()?;

    Ok(providers[selection].0.to_string())
}

pub fn prompt_api_key(provider: &str) -> Result<String> {
    let key = Password::with_theme(&ColorfulTheme::default())
        .with_prompt(format!("Enter {} API key", provider))
        .interact()?;
    Ok(key)
}

pub fn confirm_delete(name: &str) -> Result<bool> {
    let confirmed = Confirm::with_theme(&ColorfulTheme::default())
        .with_prompt(format!("Delete {}? This cannot be undone.", name))
        .default(false)
        .interact()?;
    Ok(confirmed)
}
```
